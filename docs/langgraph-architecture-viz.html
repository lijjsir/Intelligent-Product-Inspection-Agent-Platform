结构问题分析

🔴 严重：2 个节点不可达
supervision_sampling_agent 和 lab_detection_agent 已在图中注册但没有任何入边， manager_route_policy 没有连接到它们，永远不会被执行。
🟡 中等：专业 Agent 节点全是占位
market_monitor、public_opinion、trend_evolution 等节点返回硬编码的空 candidate_memories， 只有 quality_judgement 有实际逻辑。
🟡 中等：子图集成非 LangGraph 原生方式
MemoryManagerGraph 没有使用 add_node(graph) 方式集成子图， 而是手动调用异步函数。QualityJudgementSubgraph 只是一个 Python 类包装器。
🟡 中等：路由策略单一
select_subgraph() 始终返回 quality_judgement，三种路由模式 (legacy_only / canary_non_pdf / router_enabled) 最终都指向同一目标。
🔵 注意：两套并行系统
InspectionGraph (顺序管道) 和 MemoryManagerGraph (StateGraph) 互不通信， 分别处理 pipeline 和 chat 入口，代码路径完全独立。
🔵 注意：拓扑目录与运行时不一致
topology_catalog 声明 7 个子图归属 MemoryManagerGraph，但只有 2 个活跃 (v1+)， 其余 5 个为 v0 占位。

总体评价

设计意图 监督者-工作者 (Supervisor-Worker) 模式，MemoryManager 作为中央调度器
当前状态 MVP 阶段 — 核心链路可用，多 Agent 扩展骨架已搭建但未实现
合理之处 ✅ Supervisor-Worker 模式适合多 Agent 协作
✅ 记忆生命周期（候选→验证→治理）设计完整
✅ 质量门控 + 污染监控的防御性设计
✅ QualityChatGraph 顺序管道简洁清晰，适合 QA 场景
待改进 ❌ 2 个不可达节点应补全路由或移除
❌ 5 个 v0 占位子图要么实现要么清理
❌ 路由策略需要真正的多路分发逻辑
❌ 两条入口 (chat/pipeline) 应该统一到一个监督图
❌ 应使用 LangGraph 原生 subgraph add_node 方式集成
