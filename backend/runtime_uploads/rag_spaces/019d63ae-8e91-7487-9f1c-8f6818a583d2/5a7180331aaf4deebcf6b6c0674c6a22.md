  目标
  在 ops/analytics 下保留一个统一入口，页面内用 tab 区分：

  1. 总览（现有 Analytics 主体）
  2. 质量报告（现有 QualityReport 主体）
  3. 质量追踪（保持独立或后续再并入）

  设计原则

  1. 先前端合并，后端接口不动，降低风险。
  2. 路由先兼容再收敛，避免历史链接失效。
  3. 时间筛选统一，避免用户在两个页面重复设置。

  计划中的文件改动（方案）

  1. 新增统一 store（可选先不做，直接在 view 内组合也行）

  - 新建 frontend/src/stores/analytics-unified.store.ts
  - 职责：
      - 管理 activeTab、dateRange
      - 并行拉取 analyticsApi.getOverview + qualityApi.getReport
      - 对外暴露 overview/report/loading/error
  - 关键接口：
      - fetchAll(params?)
      - setTab(tab)
      - setDateRange(range)

  2. 合并页面组件

  - 改造 frontend/src/views/AnalyticsView.vue
  - 调整为“壳页面 + 内部分区渲染”
      - 顶部保留 hero 和统一时间筛选
      - 中部改为 el-tabs（overview / quality）
      - overview 面板放原 Analytics 内容
      - quality 面板迁入原 QualityReport 内容
  - 为减少文件膨胀，建议拆两个子组件：
      - frontend/src/components/business/analytics/AnalyticsOverviewPanel.vue
      - frontend/src/components/business/analytics/QualityReportPanel.vue

  3. 路由兼容与收敛

  - 修改 frontend/src/router/routes/ops.routes.ts
  - 方案：
      - 保留 ops-analytics 指向合并后页面
      - 将 ops-analytics-report 改为 redirect 到 /ops/analytics?tab=quality
      - ops-analytics-tracing 暂时保留不变
  - 示例策略：
      - 老路径仍可访问
      - 新入口统一为 /ops/analytics

  4. Tab 导航组件调整

  - 修改 frontend/src/components/business/analytics/AnalyticsTabNav.vue
  - 从“跨路由切页”改为：
      - 页面内 tab：总览/质量报告
      - 跨路由项：质量追踪（点击跳路由）
  - 或者保留现状，把“总览/质量报告”切换迁入 AnalyticsView 内部控制。

  5. 旧页面处理

  - frontend/src/views/quality/QualityReportView.vue
  - 两阶段：
      1. 第一阶段保留文件，仅用于 redirect（最稳）
      2. 第二阶段确认无引用后删除

  状态与交互细节

  1. 统一 loading：

  - 首屏并行加载两个接口；tab 切换不重复请求（除非主动刷新）

  2. 统一筛选：

  - 时间范围一次设置，两块数据同时刷新

  3. 错误展示：

  - overview 和 quality 分区独立报错，不互相阻塞

  4. URL 同步：

  - ?tab=overview|quality，支持刷新和分享链接

  实施顺序（建议）

  1. 先完成 AnalyticsView 内合并与 UI 拆分（不改路由）。
  2. 接入 unified store 或 view 内组合请求。
  3. 再改路由 redirect 兼容旧地址。
  4. 最后清理 QualityReportView 与无效引用。

  验收标准

    1. /ops/analytics 可在一个页面查看总览与质量报告。
    2. /ops/analytics/report 自动跳转到 /ops/analytics?tab=quality。
    3. 时间筛选对两个分区都生效。
    4. 现有“质量追踪”功能不受影响。
    5. 指标值与旧页面一致（抽样核对总结果数、幻觉率、点踩率、模型对比）。

• 改动目的
  将“预警处置”从独立页面收敛到稳定性工作台内，形成“稳定性总览 + 预警处置”一体化流程；同时保留 /app/alerts 兼容跳转，避免
  中断现有入口，并补齐文档要求的 ACK / 压制 / 解决 告警闭环能力。

  实施清单

  1. 后端补齐告警处置能力

  - 在 backend/app/api/v1/alerts.py 新增 PATCH /api/v1/alerts/{alert_id}，支持 acknowledged|suppressed|resolved。
  - 保留 PUT /{id}/resolve 兼容期，内部转调新逻辑。
  - 在 backend/app/services/alert_service.py 增加统一 handle_alert，实现状态流转校验和动作落库。
  - 在 backend/app/repositories/alert_repo.py 扩展状态更新方法，按动作写入对应审计字段。
  - 在 backend/app/schemas/alert.py 将 status 收敛为枚举。

  2. 数据模型与迁移补全

  - 新增 migration，为 alert_events 增加 suppressed_by、suppressed_at、action_note（或 last_action_note）。
  - 更新 backend/app/models/alert.py 与 schema 对齐。

  3. 前端并入 Stability 的“预警”子 Tab

  - 将 frontend/src/views/AlertListView.vue 抽成可复用面板组件（如 StabilityAlertTab.vue）。
  - 在 frontend/src/views/StabilityOverviewView.vue 增加 总览 / 预警 tabs。
  - 在“预警”tab内提供三类动作：ACK、压制、解决。
  - 更新 frontend/src/api/alert.api.ts 新增 handle 方法。
  - 更新 frontend/src/stores/alert.store.ts 增加 ackAlert/suppressAlert/resolveAlert。
  - 更新 frontend/src/types/alert.types.ts 补齐状态和字段类型。

  4. 路由兼容与入口稳定

  - 在 frontend/src/router/routes/app.routes.ts 将 /app/alerts 改为 redirect 到 /app/stability?tab=alerts。
  - 保留 app-alerts 路由名用于兼容历史链接/埋点。
  - 在 StabilityOverviewView 中实现 query.tab 同步，支持刷新和分享。

  5. 查询与可操作性增强

  - 告警列表查询补充 alert_type、关键词、时间范围（可分阶段）。

  - 后端：状态流转、权限、兼容接口回归。

  - 业务验收：满足“查看评分、ACK、解决、压制预警”的文档要求。
