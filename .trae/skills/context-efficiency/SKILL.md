# 上下文管理与大型项目高效协作技能

长上下文和复杂项目是 AI 辅助开发的两大核心挑战。本技能提供一套系统性的应对策略，让模型在任何条件下都能保持准确、高效、一致。

---

## 一、问题诊断

在开始工作前，先判断当前处于哪种困境：

| 症状 | 根本原因 | 对应策略 |
|------|----------|----------|
| 模型忘记之前的决定 | 上下文窗口压力，早期信息被挤出 | → 用**会话锚点**固定关键信息 |
| 生成代码与已有代码风格不一 | 缺乏代码上下文，模型在「猜」 | → 用**最小上下文注入**提供关键片段 |
| 前后回答矛盾 | 上下文过长，模型注意力分散 | → 开启**新会话 + 状态交接文档** |
| 复杂任务中途出错 | 任务粒度太大，依赖关系不清晰 | → **任务分解 + 检查点** |
| 网络中断后丢失进度 | 无持久化的任务状态 | → **进度快照**机制 |
| 分析大文件时遗漏细节 | 一次性读入过多信息 | → **分段读取 + 摘要压缩** |

## 二、核心策略

### 策略 A：会话锚点（Session Anchor）

**适用场景：** 对话轮次超过 15 轮，或涉及多个相互关联的决定。

在对话开头或关键节点，要求用户维护一份**锚点文档**，固定本次会话的不变前提。模型每次回答前先默读锚点。

**锚点文档格式：**

```markdown
# 会话锚点 · PIAP 后端开发

## 当前任务
为 InspectionTask 新增批量优先级调整接口（PATCH /tasks/batch-priority）

## 已确认的技术决定
- 数据库：MySQL 8.0，UUID 用 BINARY(16) 存储
- ORM：SQLAlchemy 2.0 异步模式
- 权限：批量操作要求 org_admin 及以上
- 事务：写操作必须同步写 audit_outbox

## 当前文件状态
- app/services/task_service.py ✅ 已有 create / get / list 方法
- app/repositories/task_repo.py ✅ 已有 list_by_status 方法
- app/api/v1/tasks.py ✅ 已有 GET / POST / GET:id 端点
- migrations/versions/0001_initial.py ✅ 已建表，索引齐全

## 本次不修改的文件
- app/models/task.py（ORM 结构不变）
- app/schemas/common.py（通用 Schema 不变）

## 待办清单
- [ ] task_repo.py 新增 bulk_update_priority 方法
- [ ] task_service.py 新增 batch_set_priority 用例
- [ ] tasks.py 新增 PATCH /batch-priority 端点
- [ ] 补写单元测试
```

**使用方式：** 把锚点文档粘贴在每次新消息的开头，用 `---锚点---` 和 `---正文---` 分隔。模型看到锚点后会优先以锚点为准，不会被对话历史中的噪声干扰。

### 策略 B：最小上下文注入（Minimal Context Injection）

**适用场景：** 需要模型生成与已有代码一致的新代码。

不要把整个文件贴给模型，只贴**最相关的片段**，并标注文件路径和关键信息。

**好的上下文注入示例：**

```
# 文件：app/services/task_service.py（节选，约 80 行）
# 关注点：create 方法的事务结构和 audit 写法

class TaskService(TenantAwareService):
    async def create(self, body: TaskCreate, actor_id: bytes):
        async with self._db.begin():
            obj = await self._repo.create(body.model_dump(), org_id=self._org_id)
            await AuditService(self._db).log(
                actor_id=actor_id,
                resource_type="task",
                resource_id=obj.id,
                action="create",
            )
        await queue.push('tasks:pending', bin_to_uuid(obj.id))
        return TaskResponse.model_validate(obj)

# 请照此风格，新增 batch_set_priority 方法
```

**不好的做法：** 把 500 行的完整文件全部粘贴，让模型自己找相关部分。信息越多，注意力越分散，错误率越高。

**注入优先级（由高到低）：**
1. 被修改函数的上下文（±20 行）
2. 该文件的类定义和 `__init__`
3. 直接依赖的基类或接口
4. 同层其他文件的同名方法（作风格参考）

### 策略 C：新会话 + 状态交接文档（Session Handoff）

**适用场景：** 对话已经很长，模型开始出现前后矛盾；或网络中断需要重启会话。

在旧会话中让模型生成一份**状态交接文档**，然后在新会话开头粘贴它。

**让模型生成交接文档的指令：**

> 请为我生成一份状态交接文档，包含：当前任务目标、已完成的内容摘要、未完成的部分、关键技术决定、下一步的具体操作。格式参考 Session Handoff 模板。

**Session Handoff 模板：**

```markdown
# 状态交接文档
生成时间：{datetime}
原会话轮次：约 {N} 轮

## 任务目标
{一句话描述本次开发目标}

## 已完成
- {文件名}：{完成了什么，关键方法/接口名}
- ...

## 未完成（按优先级排序）
1. {最紧迫的未完成项，描述清楚期望输出}
2. ...

## 关键技术决定（不可推翻）
- {决定1}：{理由}
- {决定2}：{理由}

## 已知问题 / 待确认
- {问题描述}：{当前状态或假设}

## 新会话第一步
{告诉新会话的模型应该做什么，越具体越好}
```

**新会话开头的标准 prompt：**

```
以下是上一个会话的交接文档，请先阅读并确认理解后，再继续工作：

[粘贴交接文档]

确认后，请执行「新会话第一步」中的任务。
```

### 策略 D：任务分解 + 检查点（Task Decomposition）

**适用场景：** 需要跨越多个文件、多个层次的复杂任务（超过 3 个文件同时修改）。

将大任务拆成**有序的原子步骤**，每步完成后设置检查点，确认无误再进行下一步。

**分解模式：**

```
复杂任务
├── Step 1：数据层（ORM + Migration）    ← 无依赖，先做
│   └── 检查点：表结构是否正确？索引是否声明？
├── Step 2：类型层（Schema + Types）     ← 依赖 Step 1
│   └── 检查点：字段名是否与 ORM 对齐？
├── Step 3：数据访问层（Repository）     ← 依赖 Step 1/2
│   └── 检查点：查询方法是否命中正确索引？
├── Step 4：业务逻辑层（Service）        ← 依赖 Step 3
│   └── 检查点：事务边界是否正确？Audit 是否写入？
└── Step 5：接口层（API Route）          ← 依赖 Step 4
    └── 检查点：路由是否注册？权限守卫是否正确？
```

**检查点指令模板：**

> Step {N} 已完成。请在继续之前：
> 1. 列出本步骤生成/修改了哪些文件和方法
> 2. 确认是否有未处理的依赖（import、路由注册等）
> 3. 如果有问题，现在修复；没问题就回复「Step {N} ✅，可以继续」

---

### 策略 E：分段读取 + 摘要压缩（Chunked Reading）

**适用场景：** 需要分析或理解大型代码文件（>200行）或多个文件。

不要一次性让模型读取整个代码库，按以下顺序**分段注入**：

```
第一段：接口定义（函数签名、类定义、公开方法）
         → 模型建立结构图
第二段：核心业务逻辑（最复杂的 1-2 个方法）
         → 模型理解主流程
第三段：边缘处理（错误处理、校验、日志）
         → 模型补全细节认知
```

**压缩已理解内容的指令：**

> 你已经分析了 task_service.py 的全部内容。请用不超过 200 字总结它的核心职责、主要方法和依赖关系，作为后续对话的压缩摘要。

把这个摘要保存到锚点文档里，替换原始代码，节省上下文空间。

## 三、上下文预算管理

把当前会话的上下文想象成一个有限的「注意力预算」。以下是各类内容的预算消耗估算：

| 内容类型 | 预算消耗 | 优化建议 |
|----------|----------|----------|
| 完整代码文件（500行） | 高 | 只贴相关片段（±30行） |
| 数据库 DDL 建表语句 | 中 | 只贴表名+关键列+索引名 |
| 错误堆栈 | 中 | 只贴最后 20 行 + 第一行异常 |
| 需求描述（自然语言） | 低 | 可以完整保留 |
| 已完成任务的对话记录 | 高 | 用交接文档替代，开新会话 |
| 锚点文档 | 低 | 每轮携带，值得 |

**预算告警信号（出现以下情况时主动开新会话）：**
- 模型开始重复之前已经解决的问题
- 生成的代码引入了明显不属于本项目的写法
- 模型说「根据你之前说的…」但描述的内容是错误的
- 同一个问题，模型在不同轮次给出了不同答案

## 四、网络中断恢复流程

网络中断或会话意外终止时，按以下步骤快速恢复：

```
Step 1  新建会话
Step 2  粘贴锚点文档（如果有）
Step 3  描述中断前的最后状态：
        「上次生成到 task_service.py 的 batch_set_priority 方法，
          代码已经生成但还没有确认是否正确，请重新生成该方法」
Step 4  如果有中断前生成的代码，把它贴出来：
        「这是中断前生成的代码，请检查是否有问题并继续」
Step 5  继续从检查点恢复
```

## 五、针对 PIAP 项目的专项建议

### 5.1 模块边界速查（减少跨模块混淆）

当任务涉及多模块时，先用这张表确认边界，不要跨层传递职责：

```
用户问题 → 判断所在层 → 只修改该层文件

「端点返回 500」     → 先查 error_handlers.py，再查 Service
「数据没查出来」     → 先查 Repository 的过滤条件，确认 org_id 注入
「稳定性分数不对」   → 查 agent/stability/dimensions/，确认权重
「SSE 消息丢失」     → 查 agent/graph/nodes/ 的 node_logs 追加逻辑
「前端数据不更新」   → 查 Pinia Store 的乐观更新逻辑
「权限拦截失效」     → 查 app/core/permissions.py 的 RBAC 矩阵
```

### 5.2 高频修改文件的精简注入模板

以下是 PIAP 项目中最常被修改的文件，每个附带最小上下文注入格式：

**注入 TaskService 上下文时：**
```
# task_service.py 核心结构（只需这些）
class TaskService(TenantAwareService):
    def __init__(self, db, org_id): ...
    async def create(self, body, actor_id): ...  # 事务 + audit 模式
    async def get(self, id): ...                  # NotFoundError 模式
    async def list(self, query): ...              # 分页模式
# 新增方法请与 create 保持相同的事务和 audit 风格
```

**注入 Pinia Store 上下文时：**
```
# task.store.ts 核心结构（只需这些）
export const useTaskStore = defineStore('task', () => {
  const items = ref<InspectionTask[]>([])
  const total = ref(0)
  async function fetchTasks(query) { ... }   // loading + try/finally 模式
  async function createTask(payload) { ... } // 乐观更新模式
  return { items, total, fetchTasks, createTask }
})
# 新增 Action 请与 fetchTasks 保持相同的 loading 和错误处理风格
```

### 5.3 风险最高的跨层操作清单

以下操作最容易因上下文不足而出错，执行前务必注入对应上下文：

| 操作 | 必须注入的上下文 |
|------|-----------------|
| 新增 API 端点 | router.py 现有注册列表 + deps.py 签名 |
| 新增 ORM 关联关系 | 两个模型文件的完整列定义 |
| 修改 LangGraph 图结构 | inspection_graph.py 的节点注册和边定义 |
| 新增稳定性维度 | scorer.py 的权重汇总逻辑 |
| 修改 Pinia Store | 所有引用该 Store 的 View 的引用方式 |

## 六、快速指令卡

以下是可以直接复制使用的指令模板：

**生成会话锚点：**
> 请根据我们目前的对话，生成一份会话锚点文档，包含当前任务、已确认的技术决定、文件状态和待办清单。

**生成状态交接文档：**
> 我们的对话已经比较长了，请生成一份状态交接文档，让我可以在新会话中无缝继续工作。

**分解当前任务：**
> 请把「{任务描述}」分解为有序的原子步骤，列出每步涉及的文件、依赖关系和检查点，我们逐步执行。

**压缩已理解代码：**
> 你已经分析了 {文件名}，请用 150 字以内总结它的核心职责和主要方法，作为后续对话的压缩摘要替代原始代码。

**恢复中断任务：**
> 我们的会话中断了。上次的进度是：{描述中断前状态}。请从这里继续，先确认你理解当前状态，再执行下一步。

**诊断前后矛盾：**
> 你在第 {N} 轮说了 {A}，但刚才说了 {B}，这两个是矛盾的。请重新确认正确答案，并说明为什么出现了这个矛盾。
