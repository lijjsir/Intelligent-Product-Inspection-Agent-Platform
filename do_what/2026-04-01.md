# 2026-04-01 工作总结

## 目标概述

本轮工作围绕“质量检测聊天智能体”展开，目标是基于现有项目实现一个新的 `LangGraph` 聊天子图，并把它接入前端聊天菜单界面，支持用户发消息提问、智能体流式回答、任务信息补充、RAG 空间接入，以及相关的运行稳定性、编码稳定性和权限问题修复。

## 已完成的核心实现

### 1. 新增质量检测聊天智能体子图

- 新建独立的 `LangGraph` 聊天子图，作为聊天页的执行内核，不复用旧的质量检测执行子图。
- 子图面向标准解读、缺陷说明、流程咨询、检测结果解释等问答场景。
- 子图包含的主要能力：
  - 输入归一化
  - 历史会话加载
  - 意图识别
  - RAG 检索
  - 推理与结构化回答
  - 质量门禁
  - SSE 流式输出
  - 最终消息落库
- 新聊天链路已接入后端路由与服务层。

### 2. 聊天意图分流与任务补全流程

- 在聊天子图中加入了意图分流，不再把所有输入都强制按质量问答处理。
- 当前支持的主要意图：
  - `smalltalk`
  - `quality_qa`
  - `task_create`
  - `task_followup`
- `smalltalk` 不再触发 RAG 和 embedding，适合“你是谁”“你能做什么”这类普通对话。
- `quality_qa` 继续走知识检索与质量门禁。
- `task_create` / `task_followup` 支持从对话中提取任务字段，并识别缺失字段。
- 当前缺失必要信息时，前端会弹出任务表单，用户填写后可以直接创建任务。

### 3. 聊天域持久化与 SSE 通道

- 新增聊天会话和消息落库能力。
- 已支持：
  - 会话创建
  - 会话列表
  - 历史消息加载
  - 发送消息
  - 流式订阅
  - 任务结果回写
- SSE 已改为短期 `stream token` 鉴权。
- 聊天 SSE 已加入心跳保活，避免连接空闲时被浏览器或代理误判断开。

### 4. 聊天前端交互修正

- 发送消息时，用户消息会立即本地显示在聊天框中。
- 发送后输入框会立刻清空。
- 助手消息顺序稳定地显示在用户消息之后，不再出现消息上下颠倒。
- 支持附件上传并在输入区以 chip 形式显示。
- 支持任务补充按钮和表单弹窗。
- 任务创建成功后，聊天中会显示任务结果和查看任务入口。

### 5. RAG 空间能力接入

- 新增 `RAG 空间` 管理能力。
- 保持双存储分层设计：
  - Qdrant 负责向量、文档分块和检索
  - MySQL 负责 RAG 空间元数据、文件清单、创建人、启用次数、组织隔离
- 新增能力包括：
  - RAG 空间列表
  - 创建 RAG 空间
  - 上传文档到 RAG 空间
  - 聊天时选择当前会话使用的 RAG 空间
- 前端布局已调整：
  - `RAG 空间` 变为独立菜单，放在聊天菜单下方
  - 聊天页主要空间还给消息区域
  - 历史会话 / 新增会话 / 删除会话移动到顶栏靠近 `PIAP 控制台`

## 重要排障与修复

### 1. Langfuse / Pydantic warning 修复

曾出现两类告警：

- `PydanticSerializationUnexpectedValue`
- `No active span in current context`

已修复思路：

- 不再把不稳定的原始 `usage` dict 直接塞进严格类型字段
- 只保留稳定 token 计数
- 避免在无 active span 的上下文中读取 observation id

这些问题属于观测链路告警，不是业务逻辑错误，但已进行了修正，减少日志污染。

### 2. 聊天页无法连接后端

已确认前后端基本通信正常，问题主要来自以下几类：

- 本机同时存在多个残留后端实例，旧实例没有新聊天接口
- 登录组织或账号参数错误导致 `403`
- SSE 流长时间空闲导致会话中断误报

处理结果：

- 通过单实例启动排除了旧后端干扰
- 登录链路已验证可正常获取 token
- 聊天 SSE 已增加心跳保活

### 3. `rag_spaces` 缺表报错

报错内容：

- `Table 'piap_main.rag_spaces' doesn't exist`

根因：

- 代码已升级到 RAG 空间能力
- 但数据库未完成对应迁移

已完成：

- 执行数据库迁移到 `0015_rag_spaces_and_files`
- 确认 `rag_spaces` 和 `rag_space_files` 已存在
- 在后端服务层加入更友好的错误翻译
- 当前若元数据表缺失，会返回清晰的业务提示，而不是直接把底层 SQLAlchemy 异常暴露到页面
- 前端聊天页和 RAG 页面也已改为展示明确提示，不再只显示“请求失败”

### 4. 乱码防护

你之前在根目录下加入了 `.editorconfig` 和 `.gitattributes`，但它们只能起约束作用，不能完全防止所有乱码。

已补充：

- `.vscode/settings.json`
- `scripts/dev-utf8.ps1`
- `scripts/check_mojibake.py`

用途如下：

- 强制编辑器使用 UTF-8
- 强制 Windows 终端开发时切到 UTF-8
- 扫描源码中的典型乱码片段

同时还修复和重写了若干出现过乱码的关键源码文件，确保聊天链路和提示文本可读。

### 5. `role user cannot access inspection_spec` 反复报错

根因：

- 聊天页初始化时会请求检测标准列表
- 后端权限表原先只允许 `admin` 和 `analyst` 访问 `inspection_spec`
- 普通 `user` 角色进入聊天页时就会反复触发 `403`

按要求已处理：

- 已把 `ROLE_USER` 加入 `inspection_spec` 权限集合
- 现在 `user` 角色可以访问检测标准资源
