# 2026-04-02 工作总结

## 一、总体目标

围绕 PIAP 项目完成了以下几类工作：

1. 同步并适配前后端最新开发代码，确保本地前后端和 Docker 开发链路可运行。
2. 解决聊天页常驻 SSE、登录态串页签、RAG 权限边界、任务链路断点、图片地址格式不兼容、中文乱码等一系列影响体验和稳定性的核心问题。
3. 将聊天页逐步改造成“用户提交任务 -> 智能体执行 -> 各业务页面展示对应数据”的主入口。
4. 补充 Token 统计能力，包括用户级累计 Token 汇总、聊天页展示、管理页统计表以及数据库 Alembic 迁移。

## 二、聊天与流式连接改造

### 1. 聊天流改为“按发送短连接”

已将聊天页的流式连接从“进入页面自动常驻 SSE”改为“仅发送消息时建立连接，终态后自动关闭”。

主要效果：

- 打开聊天页不再自动建立 SSE。
- 点击发送消息时才建立 SSE。
- 收到 `message_final` 或 `run_failed` 后立即关闭连接。
- 页面切换离开聊天页、登出时仍会强制关闭连接。
- 顶栏常驻“流式连接正常/中断”标签已移除。
- 仅在发送过程中显示轻量状态，例如“正在建立连接 / 智能体处理中 / 正在整理回复”。

### 2. SSE 失败兜底

如果 SSE 建连失败，但消息已成功发送：

- 前端会启动短时轮询兜底。
- 直到拉到终态消息为止。
- 避免出现“消息已发出但看不到回答”的问题。

### 3. `ECONNREFUSED /api/v1/auth/token` 归因澄清

已确认这是：

- Vite 代理到后端失败
- 原因通常是后端未启动、端口不可达或服务未监听

不是聊天流协议本身出错。

因此已把这类问题的前端提示收敛为：

- “后端连接失败”
- 不再映射为误导性的“流式连接中断”

## 三、聊天意图识别与普通问答修正

### 1. 修正“普通问答误判为质量问答”

原逻辑中：

- 只要不是 `smalltalk`
- 也不是 `task_create`
- 就默认落到 `quality_qa`

这会导致：

- `我叫tgg`
- `我叫什么名字`
- 一般闲聊或自我介绍

被错误地路由成质量问答。

现已新增独立的 `general_qa` 分支：

- 普通问答不再默认归类为 `quality_qa`
- 质量问答必须命中明显质量领域语义
- `general_qa` 不走 RAG，不走质量门控

### 2. 小对话与姓名记忆优化

为聊天图补充了“当前会话内简单记忆”能力：

- 用户说 `我叫tgg` 时，会记录为本会话的自报姓名
- 后续用户问 `我叫什么名字`
- 系统会优先从当前会话历史消息中提取最近一次姓名介绍

当前预期行为：

- `我叫tgg` -> `好的，我记住了，你叫tgg。`
- `我叫什么名字` -> `你之前告诉过我，你叫tgg。`

### 3. 任务触发词增强

已补充质量检测任务触发词：

- `质量检测`
- `质检`
- `检测任务`
- `开始检测`
- `启动检测`

这些输入会优先进入任务创建链路，而不是被当成普通质量问答。

## 四、聊天驱动任务执行闭环

### 1. 聊天页任务创建流程

已支持：

- 用户在聊天中输入任务信息
- 缺字段时展示补全表单
- 字段齐全后点击确认
- 直接创建任务并启动执行

不再要求用户再去任务详情页手动点“启动 AI 推演”。

### 2. 任务相关消息表现

聊天消息中已支持：

- 缺字段时显示“填写检测信息”
- 信息齐全时显示“确认并提交任务”
- 任务创建后在聊天里展示任务卡片
- 执行中的任务可显示阶段状态摘要

### 3. 任务执行结果回写

后端支持将任务完成或失败摘要回写到原聊天会话中。

包括：

- 任务成功的结论摘要
- 失败原因说明
- 对应任务 ID、产品编号、检测标准等关键信息

## 五、任务链路与权限边界

### 1. `role=user` 的任务访问范围

已调整为：

- `user` 只能看自己创建的任务
- 不能看其他用户的任务

### 2. `admin` 改为平台级超级管理员

当前已调整为：

- `admin` 可跨组织查看全平台所有任务
- 不再受 `org_id` 限制

这解决了：

- 不同组织下 `user` 创建的任务，`admin` 页面看不到的问题

### 3. 删除任务能力

已在任务列表页和任务详情页补上“删除任务”。

同时保留限制：

- 运行中的任务不可删除

### 4. 删除任务弹窗问题

曾出现 Element Plus 删除确认弹窗定位异常、跑到页面左上角的问题。

已从服务式弹窗改为页面内受控 `el-dialog`，后续仍需继续观察实际浏览器中的居中表现是否完全稳定。

## 六、RAG 空间、文档和权限改造

### 1. RAG 从组织共享改为用户私有

已将可见性边界由：

- `org_id`

升级为：

- `org_id + user_id`

现在行为为：

- 用户 A 创建的 RAG 空间与文档仅用户 A 可见
- 用户 B 即使同组织，也无法在列表或检索中看到 A 的知识库内容

### 2. RAG 页面结构重构

RAG 管理页已改为：

- 上半区显示空间列表
- 下半区显示当前空间的文档列表
- 增加“默认不使用文档”选项

文档展示方式由原来的卡片内零散 tag，改为更清晰的列表视图。

### 3. 文档列表与单文档单空间约束

根据最新要求，已按“一个文档就是一个空间”的方向收敛：

- 创建空间时前端只保留单文件上传
- 删除最后一个文档时，如果空间已空，则空间也会一起删除

### 4. RAG 文档持久化情况

当前 RAG 文档是持久化的，落地为三层：

1. 文件系统中的上传文件
2. 数据库中的 `rag_space_files`
3. Qdrant 中的向量索引

### 5. 删除文档能力

已补齐删除单个 RAG 文档的完整链路：

- 删除数据库记录
- 删除本地文件
- 删除 Qdrant 对应向量
- 重算空间文档数

### 6. RAG 检索位置

为了减少幻觉，当前已明确：

- 不在聊天入口盲目检索
- 不在回答最后阶段临时检索
- 而是在质量问答链路中的 `knowledge` 节点检索

并且检索过滤包含：

- `org_id`
- `user_id`
- `rag_space_id`

## 七、图片输入与结果页展示

### 1. 任务执行失败原因定位

任务执行失败的一个重要原因是：

- 聊天附件被保存为本地相对路径，如 `/uploads/...`
- 视觉模型仅接受 `base64`、`http`、`https`
- 不接受本地相对 URL

已修复为：

- 执行前自动将本地上传图片转换为 `data:` URL 再传给模型

### 2. 结果页图片不显示问题

前端开发环境曾出现结果页“图片加载失败”。

根因是：

- Vite 开发代理只转发 `/api`
- 没有转发 `/uploads`

已补上：

- `/uploads` 代理
- 结果页图片 URL 规范化处理

现在结果页中的任务分析图片已可正常加载。

## 八、登录、用户名显示和页签会话隔离

### 1. 顶栏用户名修复

原来顶部显示用户名不正确，常出现固定显示 `admin` 或残留上一用户的问题。

已修复为：

- 互不覆盖
- 一个页签登出不影响另一个页签

## 九、Docker 与 Celery 本地开发链路

### 1. Celery worker 缺失问题

项目最初只启动了依赖容器：

- MySQL
- Redis
- RabbitMQ
- MinIO
- Qdrant

没有启动：

- backend API
- celery worker

这会导致：

- 点击“启动 AI 推演”后任务进入队列
- 但没有 worker 消费
- 页面只停留在“任务已提交(celery)”

### 2. Docker 编排补齐

已新增并验证：

- `backend` 服务
- `celery-worker` 服务

同时：

- `backend/` 已使用 bind mount 挂载到容器
- 本地改动会实时映射到容器内文件系统

注意：

- `backend` 使用热重载时，代码修改可自动生效
- `celery-worker` 一般仍需要手动重启

### 3. 开发兜底

后端还补了“本地无 worker 时直接回退本地后台执行”的兜底逻辑，避免开发环境假性卡住。

## 十、中文乱码治理

### 1. 问题来源

项目中出现过多处中文乱码，原因主要是：

- Windows/PowerShell 环境下使用了错误编码读写源码
- 原本 UTF-8 中文被按 ANSI/GBK 解释后再次保存

### 2. 处理方式

已统一采用 UTF-8 命令执行规范：

- `chcp 65001`
- `$OutputEncoding = [Console]::OutputEncoding = [System.Text.UTF8Encoding]::new()`
- `PYTHONUTF8=1`
- `PYTHONIOENCODING=utf-8`

并逐步清理了：

- 登录页
- 布局页
- 聊天页
- 任务页
- 账单页等处的乱码文案

### 3. 当前状态

前后端源码范围内已清掉一批主要乱码，但后端聊天图 `graph.py` 中仍存在部分历史编码污染的正则与兜底文案，需要后续继续做一次彻底的 UTF-8 清洗。

## 十一、Token 统计与数据库迁移

### 1. 新增用户级 Token 汇总表

为了支持“知识库下拉框旁显示累计 Token”和“管理端每个用户总 Token 表”，新增了数据库表：

- `user_token_usage_summary`

用途：

- 记录每个用户累计使用的 prompt token
- completion token
- total token
- 累计成本
- 请求次数
- 最近一次使用时间

### 2. Alembic 迁移

已新增并执行 Alembic 迁移：

- `0017_user_token_usage_summary.py`

迁移内容包括：

- 为 `token_usage_ledger` 增加 `user_id`
- 创建 `user_token_usage_summary`
- 根据历史 `token_usage_ledger + inspection_tasks.created_by` 回填已有数据

### 3. 聊天页显示

已在聊天页“知识库”下拉框旁新增：

- `已使用 Token：xxxx`

### 4. 管理页展示

已在 Token 计费页新增用户维度统计表，展示：

- 用户名
- 角色
- 组织 ID
- 总 Token
- 请求次数
- 累计成本
- 最近一次使用时间

### 5. 普通问答 Token 统计补齐

之前普通问答虽然调用了模型，但未写入账本，因此不会反映在累计 Token 中。

现已修复：

- 普通问答和质量问答都会从聊天 LLM 返回的 `usage` 中提取 token
- 写入 `token_usage_ledger`
- 同时更新 `user_token_usage_summary`

当前聊天页累计 Token 会在新的普通问答或质量问答完成后自动刷新。
